      * OpenAIAdapter openAIAdapter
      * String apiKey
      * String baseUrl

      * OpeningStream? openingStream (test-only override)

    Method: Future<List<OpeningVariant>> generateOpenings({
      required String genreName,
      required String worldDescription,
      required String characterDescription,
      String? storyConcept,
    })

    Implementation follows TemplateCompletionService pattern:
    1. Build ChatMessage list:
       - System message: "你是 MuseFlow 开篇生成助手。根据给定的世界观、角色和故事概念，生成3种不同风格的开篇段落。只返回严格 JSON，不要返回 Markdown。返回格式: {\"openings\": [{\"style\": \"scene\", \"text\": \"...\"}, ...]}。每种开篇200-400字，风格鲜明：scene=场景切入以环境描写开场，character=人物切入以角色动作/心理开场，suspense=悬念切入以疑问或紧张感开场。"
       - User message: JSON with genre, world, character, optional concept
    2. Call streaming: openaiAdapter.createStream(apiKey, baseUrl, model, messages)
    3. Accumulate chunks into StringBuffer
    4. Strip markdown fences if present (```json ... ```)
    5. jsonDecode the buffer
    6. Parse 'openings' array and map to OpeningVariant objects
    7. Return List<OpeningVariant>

    Error handling:
    - Wrap in try-catch
    - On JSON parse error: throw FormatException or return empty list with error flag
    - On stream error: catch and return empty list

    OpeningStream typedef (for testing):
    - Function that takes messages and returns Stream<String>
    - In production, this is null (use real OpenAIAdapter)
    - In tests, provide mock stream

    Security constraints:
    - Enforce maxLength: storyConcept max 500 chars (truncate if longer)
    - No raw user input in system prompt (only JSON-encoded user message)
    - Structured JSON output parsing with error handling

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user→OpeningGeneratorService | Untrusted user input (storyConcept, world, character descriptions) sent to AI API |
| AI response→OpeningGeneratorService | Untrusted AI output (malformed JSON, excessive length) parsed and inserted into editor |
| editor→OpeningGeneratorSheet | Editor document state accessed for text insertion |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Tampering | Story concept input in OpeningGeneratorService | mitigate | Input length limit enforced (500 chars max, truncate), no raw user input in system prompt, structured JSON output parsing |
| T-08-07 | Spoofing | AI response returns non-JSON or malicious structure | mitigate | Try-catch around jsonDecode, graceful fallback to empty list or single-variant display, prevent crash |
| T-08-08 | Denial of Service | AI returns excessively long opening text | mitigate | Truncate each opening to max 1000 chars after parsing (not specified in RESEARCH but reasonable safeguard), or rely on prompt instruction "200-400字" |
| T-08-09 | Tampering | Editor insertion at wrong position corrupts document | mitigate | Use super_editor request system (InsertTextRequest, InsertNodeRequest) with proper selection handling, existing pattern proven in EditorAINotifier |
</threat_model>
