## RESEARCH.md Open Questions — Resolution Strategy for This Plan

**Open Question 1: GLM API Streaming Compatibility**
- Relevance: CRITICAL — this plan's Task 1 makes 30 consecutive streaming API calls. If GLM's SSE format is incompatible with openai_dart's parser, the serial generation will fail on the very first chapter.
- Resolution strategy: Before entering the 30-chapter loop, execute a single "smoke test" streaming call (1 chapter) and verify it returns content. If the smoke test fails with SSE parsing errors, stop immediately and log the error with full HTTP response details. The D-04 stop-on-error behavior applies here — do not proceed into the 30-chapter loop if streaming is fundamentally broken. Report as a blocking finding in 14-ISSUE-LOG.md.

**Open Question 2: Provider Graph Depth with Real API Credentials**
- Relevance: LOW — inherited from Plans 01/02. If Plans 01 and 02 completed successfully (container factory and AI-dependent tests passed), the provider graph is already validated. This plan reuses the same journey_container.dart.
- Resolution strategy: Inherited from Plans 01/02. No additional action needed.

**Open Question 3: Manual Spot-Check Scope Definition**
- Relevance: HIGH — this plan delivers JOURNEY-06 (manual spot-check checklist). The question is whether "knowledge injection effectiveness" can be partially automated.
- Resolution strategy: Hybrid approach. Automated checks in Task 1 (character name presence in generated content, deviation detection warnings) verify knowledge injection programmatically. The manual spot-check checklist in Task 2 focuses on UI interaction quality that cannot be automated: visual toolbar rendering, style differentiation visual quality, drag-and-drop chapter operations. The checklist explicitly separates "programmatically verifiable" items (with automated commands) from "visual/manual" items.
