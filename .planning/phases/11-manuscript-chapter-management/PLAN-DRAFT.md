# Phase 11: 文稿库与章节管理

**Status**: Planning
**Milestone**: v1.2 多文稿架构
**Goal**: 将 MuseFlow 从"单一编辑器"升级为"多文稿管理平台"，支持章节级编辑和多线程并行创作。

---

## 产品定位与核心理念

### MuseFlow 灵韵不是什么
- ❌ 不是 AI 代写打字机——禁止一键生成整本小说
- ❌ 不是小说抄袭工具——禁止硬搬热门小说套路再改名换皮
- ❌ 不是同质化流水线——区别于平台充斥的粗制滥造 AI 小说

### MuseFlow 灵韵是什么
- ✅ **匠人打磨工具**——作者亲自创作，AI 辅助润色，每行文字带"人"的温度
- ✅ **结构守护者**——通过节点功能理清脉络、人物笔记加深记忆、伏笔埋设与呼应，解决传统小说忘填坑、烂尾痛点
- ✅ **效率倍增器**——AI 快速呈现作者思路，解决拖更痛点，节省时间用于精细校对
- ✅ **多线程创作平台**——支持同时管理多部作品，在同样时间完成 2-3 本小说

### 核心卖点
> 生成出来的小说经过匠人（作者）亲自打磨再交由 AI 润色，同时依赖 AI 能力埋下伏笔、结尾呼应、通过人物笔记加深记忆、通过节点功能理清脉络，解决忘填坑和烂尾痛点，还能快速呈现思路解决拖更，节省时间用于精细化校对。

### 远期愿景
- 成为每个小说作者人手必备的 AI 辅助工具
- 后续开发短剧剧本 AI 辅助撰写功能，解决短剧行业剧情同质化痛点

## 背景与问题分析

### 当前架构缺陷
1. **单文稿架构**——整个应用只有一个编辑器，无法管理多部小说
2. **无章节实体**——章节只是 Fragment 的 tag，没有专用的 Chapter 数据模型
3. **编辑器是扁平文档**——所有内容在一个大编辑器中，无章节导航
4. **数据无隔离**——不同作品的知识库、故事结构混在一起

### 用户画像

#### 核心用户：有故事的"拙"笔
- **特征**：脑洞极大，世界观架构宏大，充满奇思妙想，但文字驾驭能力不足，或陷入"词不达意"的焦虑中
- **痛点**：知道发生什么，但写出来干巴巴；想写出引人入胜的开篇却无从下笔
- **Phase 11 受影响需求**：
  - CHAP-07（章节大纲/摘要）→ 解决"知道发生什么但写不连贯"——每章先写大纲确认脉络，再填充正文
  - LIB-03（快速切换文稿）→ 解决"灵感来了但要关当前文稿才能开新坑"——多线程创作，灵感不丢失
  - LIB-08（模板初始化）→ 降低新坑冷启动门槛，让"拙笔"快速进入创作状态
  - 知识库模板策略修订 → 不给现成内容避免"词不达意"时直接照搬，而是提供结构骨架让作者自己填

#### 进阶用户：追求效率的构思者
- **特征**：对故事逻辑有高要求，讨厌 AI 生成的陈词滥调和逻辑崩坏
- **需求**：需要工具辅助梳理复杂的人物关系网和剧情伏笔，而非单纯的扩写
- **Phase 11 受影响需求**：
  - CHAP-08（章节与 PlotNode 联动）→ 每章的伏笔、转折点可关联到具体章节，防止逻辑断裂
  - LIB-04（文稿独立知识库）→ 不同作品的人物关系网完全隔离，避免跨作品串人物
  - CHAP-02（章节列表面板）→ 俯瞰全局章节结构，快速发现逻辑漏洞和节奏问题
  - CHAP-01（章节状态管理）→ 草稿/初稿/定稿三级状态，支持先构思骨架再逐步精修的工作流

### 用户痛点（按画像归纳）
| 痛点 | 影响用户 | 对应需求 |
|------|---------|---------|
| 知道故事但写出来干巴巴 | 核心用户 | CHAP-07 大纲先行 + AI 润色（已有能力） |
| 多部作品人物/世界观混淆 | 进阶用户 | LIB-04 数据隔离 |
| 长篇章节多导致结构混乱 | 两类用户 | CHAP-02 章节面板 + CHAP-08 节点联动 |
| 灵感来了但要关当前才能开新坑 | 核心用户 | LIB-03 快速切换 |
| 伏笔埋了后面忘了填 | 进阶用户 | CHAP-08 章节联动 + 故事弧可视化（已有） |
| 想开新坑但冷启动门槛高 | 核心用户 | LIB-08 模板初始化 |
| 照搬模板导致同质化 | 两类用户 | 模板策略修订（套路骨架非现成内容） |

## 需求规格

### 文稿管理 (LIB)

- **LIB-01**: 用户可创建新文稿，填写标题、类型（长篇/中篇/短篇/短剧剧本）、简介
- **LIB-02**: 文稿库主页展示所有文稿卡片（封面色、标题、类型、字数、更新时间、创作状态）
- **LIB-03**: 用户可快速切换当前编辑的文稿（侧边栏或顶部 Tab）
- **LIB-04**: 每部文稿拥有独立的知识库（人物、世界观）和故事结构（PlotNode）
- **LIB-05**: 文稿支持状态管理（构思中/创作中/已完结/已停更）
- **LIB-06**: 支持文稿排序（按更新时间、按字数、按状态）
- **LIB-07**: 支持文稿归档/删除（带确认，防误删）
- **LIB-08**: 新文稿创建时可选空白或从模板初始化

### 章节管理 (CHAP)

- **CHAP-01**: 章节作为一等数据实体（Chapter），包含：标题、序号、内容、字数、状态（草稿/初稿/定稿）、创建/更新时间
- **CHAP-02**: 章节列表面板（侧边栏），支持拖拽排序、折叠/展开
- **CHAP-03**: 编辑器以章节为编辑单元——选中章节后在编辑器中加载该章节内容
- **CHAP-04**: 章节间切换无需离开编辑器——点击侧边栏章节即切换内容
- **CHAP-05**: 支持章节批量操作（标记状态、导出、删除）
- **CHAP-06**: 章节字数自动统计，文稿总字数为所有章节之和
- **CHAP-07**: 支持章节大纲/摘要字段，用于快速浏览章节梗概
- **CHAP-08**: 章节状态与故事结构（PlotNode）联动——PlotNode 可关联到具体章节

### 数据迁移 (MIGR)

- **MIGR-01**: 现有 Fragment 数据迁移到文稿/章节结构（创建"默认文稿"，所有 Fragment 作为单一章节导入）
- **MIGR-02**: 现有知识库实体按文稿隔离（默认文稿保留全部，新文稿从模板或空白开始）
- **MIGR-03**: PlotNode 的 chapter 字段关联到 Chapter 实体

## 知识库模板内容策略

### 问题
Phase 7 实现的预设模板库中，模板内容是硬编码的固定文本。这带来两个问题：
1. **版权风险**——模板内容如果过于接近现有热门小说，可能构成侵权
2. **创作束缚**——作者可能直接照搬模板内容而非发挥原创力

### 设计原则
- **只提供套路骨架，不提供现成内容**——模板给出结构框架（如"隐藏血脉→觉醒→渡劫"），但不写具体情节
- **允许参考，禁止照搬**——模板是灵感的起点，不是可上架的半成品
- **作者赋色**——让作者的天马行空想象力给传统套路赋予独特色彩，使其生动有趣或发人深思
- **保持可持续发展**——MuseFlow 的价值在于工具能力（润色、结构守护、效率倍增），不在于提供内容

### 需要调整的需求
- **TMPL-06 修订**: 模板开篇示例段落改为"结构提示"（如"开篇应交代主角身份+异常事件触发"，而非三段现成文字）
- **TMPL-05 修订**: AI 补全基于用户输入的故事概念，不从预设文本库拼接
- 新增 **TMPL-07**: 模板内容标注"灵感参考，非最终内容"提示

## 技术架构草案

### 数据模型变更
```
新增实体:
  Manuscript (文稿)
    - id, title, type(长篇/中篇/短篇/短剧), synopsis
    - status(构思中/创作中/已完结/已停更)
    - createdAt, updatedAt, lastEditedAt
    - coverColor (用于卡片展示)
    - sortOrder

  Chapter (章节)
    - id, manuscriptId(FK), number, title
    - content (AttributedString - 编辑器直接使用)
    - outline (章节大纲/摘要)
    - wordCount, status(草稿/初稿/定稿)
    - createdAt, updatedAt

修改实体:
  Fragment → 保留但降低优先级，主要用于碎片捕捉，不作为章节内容载体
  PlotNode → 增加 chapterId(FK→Chapter) 替代原来的 int chapter
  知识库实体 → 增加 manuscriptId(FK→Manuscript) 实现数据隔离

Hive Box 拆分:
  'manuscripts' → 文稿列表
  'chapters_{manuscriptId}' → 每部文稿独立章节 Box
  'knowledge_{manuscriptId}' → 每部文稿独立知识库 Box
  'plot_nodes_{manuscriptId}' → 每部文稿独立故事结构 Box
```

### UI 架构草案
```
导航层级:
  文稿库主页 (LibraryPage)
    ├── 文稿卡片网格/列表
    ├── 新建文稿按钮
    └── 点击文稿 → 进入文稿工作区

  文稿工作区 (ManuscriptWorkspace) [StatefulShellRoute]
    ├── 侧边栏: 章节列表 + 知识库入口 + 故事结构入口
    ├── 主区域: 编辑器 (以当前章节为编辑单元)
    ├── 顶部: 文稿标题 + 切换文稿快捷入口 + 返回文稿库
    └── 底部: 状态栏 (总字数、当前章节状态)

  侧边栏章节面板:
    ├── 章节列表 (可拖拽排序)
    ├── 每章: 序号 + 标题 + 字数 + 状态标签
    ├── 新增章节按钮
    └── 章节右键/长按菜单 (重命名、改状态、删除)
```

### 状态管理
```
ManuscriptProvider (AsyncNotifier)
  - 当前文稿状态
  - 文稿列表 CRUD
  - 切换文稿

ChapterListProvider (AsyncNotifier, 依赖 ManuscriptProvider)
  - 当前文稿的章节列表
  - 当前选中章节
  - 章节排序、CRUD

EditorChapterProvider (AsyncNotifier, 依赖 ChapterListProvider)
  - 当前编辑章节的 Document
  - 保存/自动保存
```

## 成功标准

1. 用户可以在文稿库中创建、管理、切换多部文稿
2. 每部文稿拥有独立的章节列表、知识库、故事结构
3. 编辑器以章节为编辑单元，切换章节无需离开编辑器
4. 章节支持排序、状态管理、字数统计
5. 现有数据平滑迁移到新架构
6. 知识库模板改为套路骨架而非硬编码内容
7. 模板内容标注"灵感参考，非最终内容"
