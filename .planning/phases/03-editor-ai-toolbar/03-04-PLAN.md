    The `SelectionLayerLinks.expandedSelectionBoundsLink` provides a `LeaderLink` whose `offset` can be read to determine the selection's position. However, the LeaderLink's offset is in the document coordinate space, not the viewport.

    **Approach: Use MediaQuery or LayoutBuilder for viewport detection**

    1. Wrap the `Follower.withOffset` in a `LayoutBuilder` to get the available height
    2. The selection bounds leader is positioned by `SelectionLeadersDocumentLayer` at the selection's visual position
    3. Read the leader's `offset` from `widget.selectionLayerLinks.expandedSelectionBoundsLink.offset` to get the Y position
    4. If `leaderY > viewportHeight * 0.6` (bottom 40%), flip: use `leaderAnchor: Alignment.topCenter, followerAnchor: Alignment.bottomCenter` with negative Y offset
    5. Otherwise: use the current `leaderAnchor: Alignment.bottomCenter, followerAnchor: Alignment.topCenter`

    **Implementation:**

    ```dart
    Widget _buildFollower({required Widget child}) {
      return LayoutBuilder(
        builder: (context, constraints) {
          // Get the leader's Y position to determine if we should flip
          final leaderOffset = widget
              .selectionLayerLinks.expandedSelectionBoundsLink.offset;
          final viewportHeight = constraints.maxHeight;
          final isInBottomHalf = leaderOffset.dy > viewportHeight * 0.6;

          return Follower.withOffset(
            link: widget.selectionLayerLinks.expandedSelectionBoundsLink,
            leaderAnchor:
                isInBottomHalf ? Alignment.topCenter : Alignment.bottomCenter,
            followerAnchor:
                isInBottomHalf ? Alignment.bottomCenter : Alignment.topCenter,
            offset: Offset(0, isInBottomHalf ? -8 : 8),
            boundary: const ScreenFollowerBoundary(),
            child: child,
          );
        },
      );
    }
    ```

    Note: The `LeaderLink.offset` property may not be directly accessible. If the API doesn't expose the offset, use `MediaQuery.of(context).size.height` combined with a `GlobalKey` on the Follower to estimate position. Alternatively, use the `Follower` widget's built-in boundary detection -- the `ScreenFollowerBoundary` already prevents the toolbar from going off-screen, which partially addresses the flip need.

    If `LeaderLink.offset` is not accessible, a simpler approach: use `Follower.withAligner` with a custom `FollowerAligner` that checks the leader position and decides alignment dynamically. Or: wrap in a `Builder` that reads `MediaQuery` and positions accordingly.

    **Simplest viable approach:** If the LeaderLink offset API is not available, use the `ScreenFollowerBoundary` (already in place) to prevent off-screen rendering, and add a `ValueListenableBuilder` on the selection that computes the approximate viewport position from the document node index. The toolbar will naturally stay visible thanks to the boundary -- the flip is an enhancement, not a blocker.
